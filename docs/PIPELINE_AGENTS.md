# Pipeline Agents: Fetch & Extract Strategy

## 1. Fetch Agent (논문 수집)

### 원칙

- **너무 많은 양을 한 번에 받지 않는다.** 한 번에 가져오는 논문 수를 제한하고, 품질을 확인한 뒤 점진적으로 확장한다.
- **스마트하게 키워드를 조금씩 확장·조정**한다. 처음에는 좁은 키워드(또는 소수 저널·짧은 기간)로 시작하고, 검증 후 키워드·저널·기간을 단계적으로 넓힌다.

### Human studies only (사람 대상만)

- **동물 모델 연구는 사용하지 않는다.** 논문 수집은 **사람(human) 대상 연구로 한정**한다.
- 검색 쿼리에 `"humans"[MeSH Terms]`를 넣어, 인덱싱된 논문 중 인간 연구로 태깅된 것만 가져온다. config: `fetch.humans_only: true` (기본값).

### Volume (양 제한)

- **max_results**: 한 번의 실행에서 가져올 최대 논문 수. 초기에는 작게(예: 5~10), 검증 후 20, 50 등으로 조정.
- **days_back**: 검색 기간. 초기에는 짧게(예: 30일), 이후 90일·1년으로 확장.
- **skip_existing**: 이미 받은 PMCID는 건너뛰므로, 반복 실행 시 **새 논문만** 추가된다. 한 번에 많이 받지 않고 여러 번에 나눠 받을 수 있다.

### Smart keyword expansion (키워드 단계적 확장)

- **topic_keywords** (config): 검색 시 제목/초록에 적용할 주제 키워드. 비어 있으면 “저널 + 기간 + open access”만 사용(가장 넓은 검색).
- **초기(seed)**: 키워드를 좁게 — 예: `["nutrition", "diet"]` → 음식·영양 관련 논문 위주.
- **확장 1**: 키워드 추가 — 예: `["nutrition", "diet", "aging", "cardiovascular"]`.
- **확장 2**: 키워드 더 추가하거나, `topic_keywords`를 비우면 저널·기간 내 전체로 확장.
- 매 단계마다 **품질(추출 결과, KG 상태)을 확인**한 뒤 다음 단계로 넘긴다.

### Config 요약

- `fetch.max_results`: 한 번에 받을 최대 논문 수 (작게 유지 권장).
- `fetch.days_back`: 검색 기간.
- `fetch.journals`: 저널 목록 (필요 시 단계적으로 추가).
- `fetch.topic_keywords`: (선택) 주제 키워드 리스트. 비우면 키워드 필터 없음. 조금씩 확장·조정.

## 2. Extract Agent (triple 추출)

### 원칙

- **가능한 비싸지 않은 모델부터** 사용한다. 비용을 낮게 유지한 채로 파이프라인을 돌리고, 품질을 확인한다.
- **정확도를 꼭 확인**한 뒤, 필요할 때만 더 비싸고 정확한 모델로 옮긴다. **비용과 정확도의 밸런스**를 의식적으로 맞춘다.

### Model tiers (모델 단계)

| Tier | 예시 모델 | 용도 | 비용 |
|------|------------|------|------|
| **저비용** | gemini-2.0-flash-lite | 기본 추출, 대량 처리, 초기 검증 | 낮음 |
| **중간** | gemini-2.0-flash | 품질 이슈 보완, 일부 논문 재추출 | 중간 |
| **고비용/고정확** | gemini-2.0-pro 등 | 샘플 검증, 골드 세트, 중요 논문만 | 높음 |

- **시작**: config에서 **저비용 모델**(예: `gemini-2.0-flash-lite`)을 기본으로 사용.
- **정확도 확인**: 샘플 논문으로 추출 결과를 사람이 검토하거나, predicate/entity 타입 오류 비율을 측정. 문제가 있으면 같은 논문을 **중간/고비용 모델**로 재추출해 비교.
- **전환 기준**:  
  - 저비용으로 대부분 만족스러우면 그대로 유지.  
  - 특정 유형의 오류가 많으면, 해당 유형만 중간·고비용 모델로 처리하거나, 전체를 중간 모델로 올린다.  
  - “조금씩” 올리며 비용 증가와 정확도 개선을 비교해 **밸런스를 맞춘다**.

### Config

- **extract.model**: 사용할 모델 이름 (예: `gemini-2.0-flash-lite`). 여기서 모델을 바꾸면 비용·정확도 트레이드오프를 조정할 수 있다.
- (선택) 나중에 **논문별/배치별로 다른 모델**을 쓰는 정책(예: 고영향 논문만 pro)을 도입할 수 있다.

### 검증 제안

- 주기적으로 **추출 결과 샘플**을 사람이 검토 (subject/object 타입, predicate가 스키마와 일치하는지).
- 오류 유형별 비율을 기록하고, 모델 전환 전·후로 비교해 **정확도 개선 vs 비용**을 평가.
